Klaviyo Integration — Testing Instructions

For the Klaviyo OAuth review team · Last updated: June 25, 2026

This document walks you through testing the Trainer Launch OAuth integration with Klaviyo end-to-end. The integration lets fitness coaches connect their Klaviyo account to Trainer Launch so we can publish AI-generated email campaigns, templates, and lists to their account on their behalf.

Test environment: The production app at https://trainerlaunch.app. No separate staging environment — Trainer Launch is a single environment SaaS product. Test accounts can sign up freely; we do not require a payment method.

1. About Trainer Launch

Trainer Launch is an AI-powered marketing platform for online fitness coaches and creators. From a short intake form, we generate a complete launch kit (sales page, email sequence, ad copy, scripts, video scripts, ManyChat DM flow, launch plan, ad graphic templates) personalised to the coach's brand and offer.

The Klaviyo integration lets coaches publish the generated email sequence and audience lists to their own Klaviyo account in one click, rather than copy-pasting between platforms.

2. Test account setup

  1. Visit https://trainerlaunch.app and click Start your launch (or any "Sign up" button)
  2. Enter any valid email address you control and a password
  3. You'll receive a confirmation email from kelsey@trainerlaunch.app (sent via Resend, authenticated via SPF/DKIM/DMARC for trainerlaunch.app). Click the link to confirm.
  4. Confirmation redirects you to https://trainerlaunch.app/dashboard.html with your session already established

You'll also need a Klaviyo account in any tier — a free trial account is sufficient for testing the OAuth flow.

3. Install flow — connecting Klaviyo

  1. From the dashboard, scroll to the Connections section
  2. You'll see four platforms listed: Klaviyo, Meta Ads, Google Ads, Duda. Only Klaviyo is currently enabled — the others show "Coming soon."
  3. Click Connect next to Klaviyo
  4. You will be redirected to https://www.klaviyo.com/oauth/authorize?... with the standard Klaviyo consent screen
  5. The consent screen will list the scopes Trainer Launch is requesting:
    • campaigns:read, campaigns:write
    • lists:read, lists:write
    • templates:read, templates:write
    • profiles:read, profiles:write
  6. Click Allow to authorize the connection
  7. Klaviyo will redirect you to https://trainerlaunch.app/api/oauth/klaviyo/callback?code=...&state=...
  8. Our server validates the signed state token, exchanges the code (with PKCE S256) for an access + refresh token, stores those tokens in an Anthropic Managed Agents vault (encrypted at rest), and records the connection in our database
  9. You are redirected to https://trainerlaunch.app/dashboard.html?connect=ok&platform=klaviyo
  10. A green success banner appears: "Connected to Klaviyo."
  11. The Connections panel updates to show Klaviyo as Connected, with a Disconnect button replacing the Connect button

4. Core functionality — what the integration does

After connection, the Trainer Launch dashboard becomes able to publish generated content to the coach's Klaviyo account. The Phase 1 publish flow (currently in development at time of OAuth review) will:

  • Create email templates in the coach's Klaviyo (uses templates:write) — one per generated email in the launch sequence
  • Create or update a list for the launch audience (uses lists:read, lists:write)
  • Create campaigns as drafts on the coach's behalf, scheduled to send at the times in the generated launch plan (uses campaigns:read, campaigns:write)
  • Read profiles for segment validation and basic audience analytics inside Trainer Launch (uses profiles:read)
  • Update profile attributes for tagging when a coach manually imports their CRM during onboarding (uses profiles:write)

Every Klaviyo write action happens only at the coach's explicit instruction (clicking Go Live after reviewing the generated content). Trainer Launch never auto-sends or auto-publishes without the coach approving the assets.

5. Uninstall / disconnect flow

  1. From the dashboard Connections section, click Disconnect next to Klaviyo
  2. A browser confirm dialog asks "Disconnect Klaviyo? You can re-connect anytime."
  3. On confirm, our server:
    • Deletes the OAuth credential from the Anthropic Managed Agents vault (revoking the stored token)
    • Deletes the platform_connections row in our database
  4. A green success banner confirms: "Disconnected from Klaviyo."
  5. The Connections panel shows Klaviyo as Not connected with a Connect button

From the coach's side, they can also revoke at any time directly from their Klaviyo account at Settings → API Keys → OAuth Apps → Trainer Launch → Revoke. Our integration handles revoked tokens gracefully: the next API call will fail, the connection status will be updated to expired, and the user will be prompted to re-connect.

6. Test scenarios

ScenarioExpected behavior
User denies consent on Klaviyo's consent screen Redirected to /dashboard.html?connect=error&platform=klaviyo&reason=access_denied with a friendly banner: "Klaviyo: You denied the authorization request."
User re-connects an already-connected Klaviyo account Overwrites the existing connection (unique constraint (user_id, platform)). The old vault credential is replaced with the new one. UI shows "Connected" with the latest connection timestamp.
State token expired (user takes more than 10 minutes to authorize) Redirected to /dashboard.html?connect=error&platform=klaviyo&reason=state_invalid with banner: "Klaviyo: Session expired during connect. Try again."
Network error during token exchange Redirected with reason=token_exchange_failed. No partial state is saved (vault credential is rolled back if downstream DB write fails).
User clicks Connect while not authenticated requireAuth() on the dashboard redirects them to /login.html first.

7. Security and data handling

  • OAuth state is HMAC-signed with a server-side secret (SHA-256 HMAC, 10-minute TTL). The state token encodes the user ID and PKCE verifier inside the signed payload — no server-side session storage required.
  • PKCE (RFC 7636, S256 method) is used for the code exchange.
  • Token storage: Klaviyo access and refresh tokens are stored in Anthropic's Managed Agents vault (encrypted at rest, auto-refreshed before expiry). They are never persisted in our application database.
  • Database storage: Only opaque references (vault ID + credential ID) are stored in our platform_connections table — never the actual tokens.
  • RLS: Every table is protected by Supabase Row-Level Security policies keyed off the authenticated user's ID.
  • Server-side environment: All token exchanges happen in Netlify Edge Functions; the Klaviyo client secret is read from a Netlify environment variable and never exposed to the browser.
