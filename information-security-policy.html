Information Security Policy

Last updated: June 25, 2026 · v1.0

This document is the canonical Information Security Policy for Trainer Launch, operated by Thryve Growth. It describes how we classify, protect, and govern information across our application, infrastructure, and operations. It consolidates and references our published policies in one place for partner reviews, security questionnaires, and audits.

1. Scope and applicability

This policy applies to all systems, data, and personnel involved in operating Trainer Launch, including:

  • Customer data (account credentials, AI-generated content, connected-platform metadata)
  • OAuth tokens granted by customers for connected platforms (Klaviyo, Meta Ads, Google Ads, Duda)
  • Source code, configuration, and build/deploy pipelines
  • Company devices, accounts, and access credentials used to operate the service

2. Roles and responsibilities

Thryve Growth's leadership is responsible for the overall information security program, including policy approval, risk acceptance, and incident escalation. Every team member is responsible for following this policy and reporting any incident or suspected vulnerability to security@trainerlaunch.app.

4. Access control

  • Customer accounts are protected by Supabase Auth (bcrypt-hashed passwords, JWT sessions, email confirmation required) and Row-Level Security policies enforced at the database layer. A customer can only access their own rows in every table — there is no application-layer carve-out.
  • Administrative access to GitHub, Netlify, Supabase, Anthropic, the registered domain, and our email provider requires multi-factor authentication. Privileged accounts are reviewed quarterly.
  • Customer OAuth tokens are stored exclusively in Anthropic's Managed Agents vault. They are never stored in our application database. Only opaque references (vault ID + credential ID) are persisted.
  • Service-role secrets are stored in Netlify environment variables, scoped to server-side edge functions, and never exposed to browser code.

5. Data protection

  • Encryption in transit: TLS 1.2 or higher on all inbound and outbound connections. HSTS is enforced at the TLD level — trainerlaunch.app is on the browser HSTS preload list, so all traffic is HTTPS-only.
  • Encryption at rest: AES-256 encryption on Supabase (managed PostgreSQL and Storage) and on Anthropic's Managed Agents vault.
  • CSRF protection: OAuth flows use HMAC-signed state tokens with a 10-minute TTL.
  • PKCE: OAuth code exchanges use RFC 7636 PKCE (S256 method).
  • Open-redirect protection: Same-origin path validation on post-auth return URLs.

6. Software development and dependency security

  • GitHub CodeQL static analysis on every commit to main.
  • Dependabot alerts and updates for vulnerable dependencies. Critical alerts are evaluated within 1 business day.
  • Secret Scanning blocks credentials from being committed.
  • Code review: All material changes are reviewed before merging to main.
  • Branch protection: Direct pushes to main are restricted.

7. Endpoint and device security

  • Full-disk encryption (FileVault / BitLocker) on all company devices
  • OS-level firewall and anti-malware enabled by default
  • Automatic OS security updates enabled
  • Screen lock on inactivity (≤ 5 minutes), strong device login + biometric authentication
  • Password manager (1Password) for all credentials — no plaintext credential storage
  • Production database credentials are never cached on developer devices; all access is via short-lived service tokens via the cloud console

8. Vulnerability management and incident response

Our published Vulnerability Disclosure Policy documents:

  • Reporting channels: GitHub Private Vulnerability Reporting and security@trainerlaunch.app
  • Acknowledgement, triage, and fix-target SLAs by severity (critical fix within 72 hours; high within 7 days)
  • Scope and out-of-scope items
  • Safe-harbor terms for good-faith research

9. Data retention, storage, and deletion

The following policies are published and binding:

  • Data Storage Policy
  • Data Deletion Policy

Customer data is deleted within 30 days of account closure or platform disconnection. OAuth credentials are revoked from the vault immediately on disconnect.

10. Vendor and third-party risk

We rely on the following providers for critical infrastructure:

  • Netlify — application hosting, edge functions, CDN
  • Supabase — PostgreSQL database, authentication, file storage
  • Anthropic — AI model API, Managed Agents vault for OAuth credentials
  • Resend — transactional email delivery
  • GitHub — source code management, security scanning
  • GoDaddy — domain registration

Each vendor maintains its own published security program; we restrict our reliance to vendors with documented controls equivalent to or stronger than our own.

11. Business continuity

Source code is version-controlled in GitHub and can be redeployed from any commit. Application data is hosted by Supabase, which provides point-in-time recovery and managed backups on paid plans. Static content is served from Netlify's CDN with rollback to any prior deployment available.

12. Logging and monitoring

Netlify provides request and function-execution logs. Supabase logs authentication events and database access. GitHub logs repository access and code-scanning alerts. Anthropic logs API usage. We retain operational logs for a minimum of 30 days.

13. Compliance and exclusions

We are committed to applicable data protection regulations including GDPR (for EU customers) and CCPA (for California customers). We do not currently hold SOC 2, ISO 27001, or HIPAA attestations. Customer data is not subject to PCI-DSS as we do not process or store payment-card data — payments are handled by upstream PCI-compliant providers.

14. Policy review

This policy is reviewed and updated at least annually, or earlier when significant changes occur to our infrastructure, threat model, or regulatory environment. Material changes are version-tagged at the top of this document.
